# 期末突击教练 —— 作业报告

---

## 一、制作动机

期末复习是每个大学生最焦虑的时刻。我自己经历过太多次这样的场景：考试还有三天，面前是一堆课件和几百页教材，完全不知道从哪里开始看。翻开书觉得自己都会，合上书又什么都想不起来。

更关键的是：**普通人复习最大的问题不是"学不会"，而是"不知道自己哪里不会"**。自己看书是被动输入，看起来很努力但效率极低。真正的有效复习应该是：先诊断薄弱点，再有针对性地突击。

市面上当然有 ChatGPT，但它是一个**被动问答工具**——你问什么它答什么。如果你自己都不知道自己哪个知识点薄弱，你怎么问？所以我想做一个**主动诊断型的 Agent**：它考你、批你、然后告诉你该复习什么。它不是一个问答机器人，而是一个教练。

这就是「期末突击教练」的出发点——用 AI 模拟一个有经验的学长，帮你做三件事：**找到弱点、制定计划、考前摸底**。

---

## 二、制作中遇到的问题和困难

### 问题 1：如何让 Agent 出高质量的题，而不是泛泛而谈？

一开始 Prompt 太简单，Agent 出的题要么太简单（"什么是 TCP？"），要么太宽泛。让 AI 出题不难，但出"能测出薄弱点的诊断题"需要精心的 Prompt 设计。

**解决方案**：在 Prompt 中明确了题目规格——题型比例、难度分级、考点标注。关键技巧是让 Agent **先看知识地图再出题**，这样题目和知识点有对应关系，批改时才能精准定位薄弱点。另外使用隐藏标记 `<!--答案:B-->` 来存储正确答案，批改时作为参考。

### 问题 2：四个阶段的上下文如何传递？

整个流程有四个阶段：课程分析 → 诊断 → 计划 → 模拟考。每个阶段都依赖上一阶段的输出（知识地图、薄弱点清单）。如果每个阶段都是独立的 API 调用，如何保证信息不丢失？

**解决方案**：设计了 MemoryStore 类作为状态管理。每个阶段的输出存入 `st.session_state`，作为下一阶段的输入参数传入 LLM。关键设计是**摘要传递而非全文传递**——把知识地图和薄弱点作为上下文注入后续 Prompt，而不是把整个对话历史塞进去，这样既节省 Token 又保证焦点不散。

### 问题 3：部署后海外 IP 被 DeepSeek 拒绝

项目部署在 Streamlit Cloud（服务器在美国），DeepSeek 官方 API 会拒绝海外 IP。这是一个实际部署中遇到的网络问题。

**解决方案**：切换到国内 API 代理平台「硅基流动（SiliconFlow）」，底层模型仍然是 DeepSeek-V3，但服务器在国内，海外访问不会被封。同时将敏感信息（API Key）通过 Streamlit Secrets 管理，不硬编码在代码中。

### 问题 4：如何让同学愿意用、容易上手？

纯命令行的工具同学不会用，太复杂的表单也劝退。需要极简的交互设计。

**解决方案**：用 Streamlit 做成了一个**向导式界面**——左边是进度条，右边是当前步骤。每一步只需要填最少的信息（课程名 + 知识点 + 时间），点一个按钮就出结果。不需要任何配置或学习成本。

## 三、技术架构

```
┌──────────────────────────┐
│   Streamlit 前端          │  四步向导式界面
│   · 课程设置 → 诊断 → 计划 → 模拟考
└──────────┬───────────────┘
           │
┌──────────▼───────────────┐
│   TutorAgent             │  五个核心方法
│   · analyze_course()     │  知识地图生成
│   · generate_diagnostic()│  出诊断题
│   · grade_diagnostic()   │  批改+薄弱分析
│   · generate_cram_plan() │  突击计划
│   · generate_mock_exam() │  模拟考试
└──────────┬───────────────┘
           │
┌──────────▼───────────────┐
│   MemoryStore            │  状态管理
│   · course_info          │
│   · knowledge_map        │
│   · weak_points          │
│   · quiz_history         │
└──────────┬───────────────┘
           │
┌──────────▼───────────────┐
│   硅基流动 API            │  DeepSeek-V3 模型
│   (OpenAI SDK 兼容)       │
└──────────────────────────┘
```

- **语言**：Python
- **前端**：Streamlit（四步向导式页面）
- **模型**：DeepSeek-V3（通过硅基流动 API）
- **核心类**：TutorAgent（业务逻辑）、MemoryStore（状态管理）

## 四、创新点

1. **主动诊断 vs 被动问答**：传统 AI 工具是你问它答，本项目是 Agent 主动出题考你，找出薄弱点后再定制计划。这是一个"诊断→计划→验证"的闭环。

2. **个性化突击计划**：不是通用的"多看看书"，而是根据诊断得分 + 剩余时间 + 知识点重要度，生成精确到小时的可执行计划。

3. **极致的用户体验**：四步向导式界面，每步一个按钮，3 分钟走完全流程。不需要写 Prompt，不需要学用法。

## 五、缺点与反思

### 当前不足

1. **题目质量依赖知识地图**：如果用户列的知识点太粗糙，Agent 出的题也会受影响。后续可以接入课程数据库，自动获取标准大纲。

2. **没有语音输入**：期末复习场景下，很多同学更愿意用说的而不是打字。加上语音转文字会大幅提升体验。

3. **诊断题数量固定**：目前是固定 4 道诊断题，没有根据课程内容量动态调整。

4. **没有长期追踪**：一次诊断只能管一门课，无法看到多门课的整体复习进度。

### 反思

这次项目让我深刻体会到：**AI 产品的价值不是"你能调用多少次 API"，而是"你能把 API 调用编排成多有用的工作流"**。

四个方法单独看，每个都是"调一次 API"。但把它们串成一条链——先分析、再诊断、再计划、再验证——就变成了一个真正的"学习教练"。这才是 Agent 的意义：不是替代人类，而是替代那些"需要专业知识才能做好的流程"。
