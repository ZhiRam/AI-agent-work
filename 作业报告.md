# 赛博室友 —— 多智能体宿舍模拟器 作业报告

---

## 一、制作动机

大学宿舍是每个人大学生活中最有记忆点的场景。四个性格迥异的人被随机分配到一起，24小时共处一个空间——这是一个天然的社会学实验场。

我一直对一个问题感到好奇：**如果把MBTI人格理论和多智能体技术结合起来，让AI模拟真实的宿舍社交动态，会发生什么？**

市面上绝大多数AI聊天产品都是"用户 ↔ AI"的一对一模式，缺乏多主体之间的自发互动。而多智能体系统（Multi-Agent System）是AI领域的前沿方向，我想用一个"人人都有共鸣"的场景来探索它——不是做冷冰冰的企业演示，而是做一个围观的"宿舍直播间"。

## 二、制作中遇到的问题和困难

### 问题1：多个Agent如何共享上下文但不共享记忆？

这是多智能体系统最核心的设计问题。如果所有Agent共享同一个对话历史和记忆，那本质上还是一个聊天机器人在扮演四个角色。**真正的多智能体要求每个Agent有独立的视角和记忆空间。**

**解决方案**：设计了两层记忆架构。短期记忆（环形缓冲区，保留最近20条）保证上下文连贯性；长期记忆（关键词重要性打分）存储重要事件。每个Agent的System Prompt中只注入**自己的**记忆，形成独立视角。同时，当一个Agent发言后，其他Agent通过`hear()`方法将这段话存入自己的记忆——"听到的"和"说出的"是两种不同的记忆类型。

### 问题2：如何让Agent之间的好感度动态变化？

对话的质量不仅体现在内容上，还体现在关系网络上。我希望Agent之间能"记仇"也能"感动"。

**解决方案**：在Prompt中约定隐藏标记语法`<!--rel:名字:±1-->`，LLM在生成回复时自主判断当前对话是否影响了对某人的好感度。后端通过正则解析这些标记并更新好感度矩阵（0-100分）。这比使用独立的情感分析模型更自然——因为"对室友的好感度变化"本身就是LLM理解对话后的一部分判断。

### 问题3：如何控制回复质量和人格一致性？

LLM有一个常见问题：聊着聊着就"崩人设"——ENFP突然说话像INTJ，或者所有Agent慢慢趋同。

**解决方案**：
- 每个人的System Prompt包含详细的人格特征、口头禅、对其他室友的态度，约800字
- Prompt中强调口语化、1-3句话、不写小作文
- 对话历史中用`{name}说：{msg}`格式明确标注发言者，防止LLM混淆身份
- Temperature设0.9保持多样性

### 问题4：对话陷入沉默或跑题怎么办？

4个Agent自由对话，有时会出现无人接话、或者话题完全发散的情况。

**解决方案**：
- 设计了30+随机事件（外卖到了、停电、出成绩…），用户可随时注入事件作为"话题引信"
- Speaker选择算法避免同一人连续发言
- 用户可以通过"附身模式"以任何Agent的身份插入话题，手动引导对话方向

### 问题5：部署和分享

同学需要访问链接来评分，但本地Streamlit只有在自己的电脑开着时才能访问。

**解决方案**：将项目部署到Streamlit Cloud（免费托管），获得一个永久链接。API Key通过环境变量或页面输入框设置，不硬编码在代码中。

## 三、技术架构

```
┌──────────────────────────┐
│   Streamlit 前端          │  用户围观对话 / 注入事件 / 附身发言
└──────────┬───────────────┘
           │
┌──────────▼───────────────┐
│   模拟引擎 (Simulator)    │  发言调度 / 上下文拼装 / 关系更新
└──────────┬───────────────┘
           │
┌──────────▼───────────────┐
│  Agent × 4  │  记忆系统   │
│ · 人格Prompt │ · 短期记忆  │
│ · 情绪状态   │ · 长期记忆  │
│ · 回复生成   │ · 好感度矩阵 │
└──────────┬───────────────┘
           │
┌──────────▼───────────────┐
│  DeepSeek API (大语言模型) │  驱动每个Agent的"大脑"
└──────────────────────────┘
```

- **语言**：Python
- **前端**：Streamlit
- **大模型**：DeepSeek（OpenAI SDK兼容）
- **核心模块**：Agent类、Simulator引擎、双层记忆系统、随机事件系统

## 四、缺点与反思

### 当前不足

1. **对话轮次长了会发散**：由于每轮只选一个Agent发言，没有显式的"对话目标"，10轮以上后话题容易漂移。后续可以考虑加入隐式的话题管理机制。

2. **好感度变化依赖LLM自觉**：标记格式是"君子协定"，LLM偶尔会忘记加标记，导致好感度更新不及时。更鲁棒的方案是用单独的judge模型来评估每轮对话的情感走向。

3. **记忆检索过于简单**：长期记忆目前按重要性和时间排序，没有用向量检索做语义匹配。当记忆量增大时，相关性会下降。

4. **没有多模态**：真实宿舍里会有图片（外卖照片、表情包）、语音（打电话），目前只有纯文本。

5. **Token消耗较大**：每轮对话需要把System Prompt + 长期记忆 + 短期记忆 + 最近对话全塞进Context，4个Agent轮流发言时Token消耗线性增长。

### 反思

这次项目让我深刻体会到：**多智能体系统最难的不是让每个Agent说话，而是让它们"记住"和"感受"**。人类社交的本质是记忆和情感的累积——昨天谁帮我拿了快递，上周谁深夜吵到我睡觉——这些跨时间的信息如何存储和召回，是整个系统从"好看"到"好用"的关键。

另外，**场景选择至关重要**。如果我做一个"企业客服多智能体"，评委不会有任何共鸣。但我选择"大学宿舍+MBTI"，每个人都有自己的宿舍故事，一秒代入。技术是为场景服务的，而不是反过来。
